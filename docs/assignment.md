# Задание на производственную практику

## 1. Общая информация

| Параметр | Значение |
|----------|----------|
| **Тема практики** | Разработка внутреннего веб-приложения с набором вспомогательных утилит для сотрудников |
| **Стек технологий** | C# (ASP.NET Core 8), JavaScript (vanilla), PostgreSQL, Docker |
| **Продолжительность** | 2 недели (10 рабочих дней) |
| **Формат работы** | Индивидуальная / парная разработка в общем репозитории |

---

## 2. Цели и задачи

### Цель практики
Разработать внутренний корпоративный веб-сайт, предоставляющий сотрудникам компании набор полезных утилит (конвертеры, генераторы, анализаторы текста и данных) через единый веб-интерфейс.

### Задачи
1. Изучить архитектуру предложенного шаблона веб-приложения (ASP.NET Core + vanilla JS).
2. Реализовать назначенные утилиты на стороне backend (C#) — бизнес-логика.
3. При необходимости дополнить frontend специфичным UI для своих утилит.
4. Обеспечить сохранение истории выполнений в базе данных (PostgreSQL).
5. Работать в системе контроля версий GitHub: ветвление, pull requests, code review.
6. Развернуть приложение в Docker (docker-compose) и продемонстрировать работу.

## 5. Требования к реализации утилиты

Каждая утилита должна:

1. **Реализовать интерфейс `IUtilityService`** (файл `Services/IUtilityService.cs`):
   ```csharp
   public interface IUtilityService
   {
       string Endpoint { get; }          // Уникальный идентификатор (напр. "multi-replace")
       string Execute(string input);     // Бизнес-логика
   }
   ```

2. **Быть зарегистрирована в `Program.cs`**:
   ```csharp
   builder.Services.AddSingleton<IUtilityService, MultiReplaceService>();
   ```

3. **Корректно обрабатывать граничные случаи** (пустой ввод, некорректные данные).

4. **Иметь описание в seed-данных** (`AppDbContext.cs`) — отметить `IsImplemented = true` после реализации.

5. **При необходимости дополнить frontend** специфичным UI (например, дополнительные поля ввода для параметров).

### Критерии оценки

| Критерий | Макс. балл |
|----------|------------|
| Работоспособность утилиты (корректная обработка всех случаев) | 30 |
| Чистота кода (читаемость, именование, комментарии) | 20 |
| Обработка ошибок и граничных случаев | 15 |
| Соответствие архитектуре (интерфейс `IUtilityService`) | 10 |
| Качество frontend (если требовался специфичный UI) | 10 |
| Наличие тестов (unit-тесты на бизнес-логику) | 10 |
| Корректность работы с GitHub (ветки, PR, история коммитов) | 5 |
| **Итого** | **100** |

---

## 6. Порядок сдачи

1. Создать ветку от `main` с именем `feature/<endpoint-утилиты>`.
2. Реализовать утилиту в backend и, при необходимости, UI во frontend.
3. Написать unit-тесты.
4. Проверить работу локально через docker-compose.
5. Создать Pull Request в GitHub, назначить ревьюера.
6. После успешного code review — слить в `main`.
7. Продемонстрировать работу руководителю практики.
