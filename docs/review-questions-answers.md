# Ответы на вопросы для проверки понимания проекта

## Блок 0 — Базовые понятия (фундамент)

### 0.1. Контейнеризация: Docker

**1. Что такое Docker? Контейнер vs виртуальная машина.**

Docker — платформа для упаковки приложения со всеми его зависимостями в изолированный «контейнер». Контейнер запускается одинаково на любой машине: у разработчика, на тестовом сервере, в продакшене.

| | Контейнер | Виртуальная машина |
|---|---|---|
| **Что изолирует** | Процесс от процесса (внутри одного ядра ОС) | Целую ОС внутри ОС |
| **Своя ОС?** | Нет, использует ядро хоста | Да, полноценная гостевая ОС |
| **Размер** | Мегабайты (наш api ~220 MB) | Гигабайты (минимум 1-2 GB) |
| **Скорость запуска** | Секунды | Минуты |

Аналогия: контейнер — отдельная квартира в многоквартирном доме (общие коммуникации, изолированное пространство). Виртуальная машина — отдельный частный дом (всё своё, но дороже).

**2. Что такое Docker-образ и контейнер? Аналогия.**

- **Образ (Image)** — шаблон, чертёж. Содержит файловую систему, код, зависимости. Создаётся командой `docker build` по инструкциям из `Dockerfile`. Read-only.
- **Контейнер (Container)** — запущенный экземпляр образа. Имеет свой read-write слой поверх образа. Можно запустить 10 контейнеров из одного образа.

Аналогия: образ — класс в программировании (описание), контейнер — объект класса (экземпляр). Или: образ — чертёж автомобиля, контейнер — собранный автомобиль.

**3. `docker-compose up`. Почему не три `docker run`?**

`docker-compose up` одной командой: создаёт общую сеть между контейнерами, запускает сервисы в правильном порядке (с учётом `depends_on` и `healthcheck`), пробрасывает порты, монтирует тома, устанавливает переменные окружения, управляет жизненным циклом: `up`/`down`/`restart`/`logs` для всех сервисов сразу.

**4. Три сервиса в `docker-compose.yml`.**

1. **postgres** — база данных PostgreSQL 16 (Alpine). Хранит список утилит и историю выполнений. Порт 5432.
2. **api** — ASP.NET Core Web API на C#. Обрабатывает бизнес-логику утилит. Порт 8080.
3. **frontend** — Nginx. Раздаёт статику (HTML/JS/CSS) и проксирует запросы к api. Порт 3000.

**5. «Пробросить порт». Зачем `5432:5432` и `3000:80`?**

Проброс порта (`ports:`) делает порт внутри контейнера доступным снаружи (на хост-машине).

- `"5432:5432"` — порт 5432 контейнера postgres → порт 5432 хоста. Чтобы подключиться к БД из DBeaver/pgAdmin.
- `"3000:80"` — порт 80 контейнера frontend (Nginx слушает на 80) → порт 3000 хоста. Чтобы открыть сайт в браузере по `http://localhost:3000`.

Без проброса: контейнеры общаются между собой через внутреннюю docker-сеть, но снаружи недоступны.

### 0.2. Веб-приложения: клиент-сервер

**6. Клиент-серверная архитектура. Кто клиент, кто сервер?**

**Клиент** — браузер (Chrome, Firefox). Отправляет HTTP-запросы, получает ответы, отрисовывает интерфейс.

**Сервер** — программа, которая слушает сеть и отвечает на запросы. У нас их два:
- **Nginx** (веб-сервер) — отдаёт статические файлы (HTML, CSS, JS)
- **ASP.NET Core** (сервер приложений) — обрабатывает API-запросы, бизнес-логику, работает с БД

Аналогия: клиент — посетитель ресторана (делает заказ), сервер — кухня (готовит и отдаёт блюдо), Nginx — официант (принимает заказ, передаёт на кухню, приносит готовое).

**7. Что такое HTTP? Какие методы?**

HTTP (HyperText Transfer Protocol) — протокол передачи данных в вебе. Работает по схеме «запрос → ответ».

| Метод | Назначение | Аналогия с БД |
|-------|-----------|---------------|
| **GET** | Получить данные | SELECT |
| **POST** | Создать новый ресурс | INSERT |
| **PUT** | Заменить ресурс целиком | UPDATE |
| **DELETE** | Удалить | DELETE |

В нашем проекте: `GET /api/utilities` — список утилит, `POST /api/utilities/sum-numbers/execute` — выполнить утилиту.

**8. URL: разбор на части.**

`http://localhost:3000/api/utilities/sum-numbers/execute`

| Часть | Значение |
|-------|----------|
| `http` | Протокол (схема) |
| `localhost` | Хост (доменное имя или IP) |
| `3000` | Порт |
| `/api/utilities/sum-numbers/execute` | Путь (path): `/api` — префикс API, `/utilities` — контроллер, `/sum-numbers/execute` — endpoint и действие |

**9. Что такое API? REST API vs обычный сайт.**

**API** (Application Programming Interface) — интерфейс для взаимодействия программ. REST API — архитектурный стиль, где данные передаются в JSON, используются стандартные HTTP-методы, URL отражает ресурс, сервер не хранит состояние клиента (stateless).

**Обычный сайт**: сервер возвращает HTML-страницу с данными внутри (смешаны данные + вёрстка).
**REST API**: сервер возвращает только данные (JSON), клиент сам решает, как их отобразить.

**10. Что такое JSON? Почему он, а не XML?**

JSON (JavaScript Object Notation) — текстовый формат обмена данными:
```json
{ "name": "Иван", "age": 25, "items": ["ручка", "блокнот"] }
```
- **Читаемый** человеком и машиной
- **Компактный** — меньше байт, чем XML
- **Нативный для JavaScript** — не нужен парсер, `JSON.parse()` встроен
- **Поддерживается всеми языками** (в C# — `System.Text.Json`)

### 0.3. Backend: C# и ASP.NET Core

**11. Что такое ASP.NET Core? Зачем фреймворк?**

ASP.NET Core — фреймворк для создания веб-приложений на C#. Предоставляет готовые «кирпичики»: маршрутизацию (`[Route]`), middleware pipeline (`UseCors()`, `UseSwagger()`), Dependency Injection, Model Binding (JSON → C#-объект), EF Core (работа с БД через C#-объекты). Без фреймворка пришлось бы писать всё это вручную — тысячи строк низкоуровневого кода.

**12. Dependency Injection «на пальцах».**

Обычный подход (без DI):
```csharp
public class UtilitiesController
{
    private AppDbContext _db = new AppDbContext(...); // жёсткая связь
}
```

С DI:
```csharp
public class UtilitiesController
{
    private readonly AppDbContext _db;
    public UtilitiesController(AppDbContext db)  // ← кто-то другой даст готовый db
    {
        _db = db;
    }
}
```

Аналогия: без DI — ты сам покупаешь продукты и готовишь. С DI — доставка еды: говоришь «хочу ужин», получаешь готовый.

**13. Singleton, Scoped, Transient. Примеры.**

| Время жизни | Создаётся | Пример из проекта |
|-------------|-----------|-------------------|
| **Singleton** | Один раз за всё время работы приложения | `UtilityOrchestrator` — хранит словарь утилит, не меняется |
| **Scoped** | Один раз на HTTP-запрос | `AppDbContext` (по умолчанию) — каждый запрос получает свой контекст |
| **Transient** | Каждый раз при запросе зависимости | Не используется в проекте |

Почему оркестратор — Singleton: его состояние неизменно после старта. Почему DbContext — не Singleton: он не потокобезопасен, два одновременных запроса испортят данные.

**14. Интерфейс (interface) в C#. Зачем?**

Интерфейс — контракт: «вот какие методы класс обязан реализовать». Сам интерфейс не содержит кода, только сигнатуры. В проекте: `IUtilityService` требует `Endpoint` и `Execute(string input)`. DI-контейнер находит все реализации через `GetServices<IUtilityService>()` и регистрирует их. Плюсы: можно заменить реализацию для тестов без изменения вызывающего кода.

**15. Контроллер (Controller). Роль.**

Контроллер — «диспетчер» HTTP-запросов. Его задача: принять запрос (роутинг), извлечь параметры (model binding), вызвать бизнес-логику (сервис, оркестратор), вернуть HTTP-ответ (200 OK с JSON, 404 NotFound). Контроллер НЕ содержит бизнес-логики — только координацию.

### 0.4. База данных

**16. Что такое база данных? Зачем она?**

База данных — организованное хранилище структурированной информации. В отличие от файла `.txt`: быстрый поиск (миллисекунды), целостность (внешние ключи), конкурентный доступ (транзакции), резервное копирование. В проекте БД хранит список утилит (27 записей) и историю выполнений.

**17. SQL. Реляционная vs нереляционная БД.**

**SQL** (Structured Query Language) — язык запросов к реляционным БД. **Реляционная** (PostgreSQL): данные — таблицы со строками и колонками, строгая схема, связи через внешние ключи. **Нереляционная** (MongoDB): данные — документы (JSON), гибкая схема. В проекте выбрана PostgreSQL — данные строго структурированы (Utility, ExecutionResult).

**18. Что такое ORM? Entity Framework Core.**

ORM (Object-Relational Mapper) — прослойка между БД и кодом. Позволяет работать с таблицами как с C#-объектами:
```csharp
var utils = await _db.Utilities.Where(u => u.Category == "Текст").ToListAsync();
```
Плюсы: защита от SQL-инъекций (параметризованные запросы), автодополнение в IDE, меньше кода.

### 0.5. Frontend: JavaScript и браузер

**19. Что такое DOM? Как JS с ним взаимодействует?**

DOM (Document Object Model) — представление HTML-страницы в виде дерева объектов. JS может находить элементы (`getElementById`, `querySelectorAll`), менять содержимое (`innerHTML`), навешивать обработчики событий (`addEventListener`). Без DOM все сайты были бы статичными, как PDF.

**20. Событие (event) и обработчик. Пример из проекта.**

**Событие** — сигнал от браузера: «кликнули», «нажата клавиша». **Обработчик** — функция, которая вызывается при событии. Пример из `utility.js`: `document.getElementById('btn-execute').addEventListener('click', async () => { ... })` — при клике отправляет запрос к backend и показывает результат. Также `Ctrl+Enter` через `keydown`.

**21. Что такое AJAX / fetch? Почему нет перезагрузки?**

**AJAX** (Asynchronous JavaScript And XML) — техника отправки HTTP-запроса из JS без перезагрузки страницы. Современная реализация — функция **fetch**. JS отправляет запрос, сервер возвращает JSON, JS обновляет нужный кусок DOM. Без AJAX: каждый запрос → полная перезагрузка → белый экран, моргание.

**22. Что такое Promise и async/await? Пример.**

**Promise** — объект, представляющий отложенное значение (pending/fulfilled/rejected). **async/await** — синтаксический сахар, делает асинхронный код похожим на синхронный:
```javascript
try {
    const data = await API.get('/utilities');
    console.log(data);
} catch (err) {
    console.error(err);
}
```
В проекте `api.js` и страницы активно используют async/await.

**23. Что такое SPA? Чем отличается от многостраничного сайта?**

**SPA (Single Page Application)**: HTML загружается один раз, JS сам обновляет содержимое. Быстро, плавно, похоже на нативное приложение. **MPA**: каждый переход → полная загрузка новой HTML-страницы. В проекте: `index.html` — единственная страница, роутер меняет содержимое `<main>` без перезагрузки.

### 0.6. Инфраструктура: Nginx

**24. Что такое веб-сервер? Зачем Nginx?**

Веб-сервер — программа, принимающая HTTP-запросы и отдающая ответы. Nginx — самый популярный веб-сервер (~34% сайтов). Зачем он, если ASP.NET Core (Kestrel) тоже умеет:
1. **Специализация**: Nginx оптимизирован для раздачи статики, Kestrel — для выполнения C#-кода.
2. **Reverse proxy**: единая точка входа, клиент общается только с Nginx.
3. **Production-фичи**: SSL, gzip, кэширование, rate limiting.

**25. Reverse proxy. Роль в проекте.**

**Reverse proxy** — посредник между клиентом и сервером. В проекте: браузер → `localhost:3000` (Nginx). Nginx смотрит на URL: `/css/`, `/js/` → отдаёт файлы сам; `/api/` → проксирует в `api:8080`; `/swagger` → проксирует в `api:8080`. Клиент не знает о внутреннем устройстве — это безопаснее.

**26. Что такое CORS? Почему блокирует?**

**CORS** (Cross-Origin Resource Sharing) — политика безопасности браузера. Браузер блокирует запросы с сайта A к API сайта B, если B явно не разрешил. Защита от: злой сайт пытается сделать запрос к вашему банку от вашего имени. В проекте `AllowAnyOrigin()` — для разработки ок, для production нужно указать конкретный домен.

**27. Swagger / OpenAPI. Польза.**

Автоматически генерирует документацию API из кода, создаёт интерактивный UI (Try It Out), экспортирует спецификацию (`swagger.json`). Фронтендеру не нужно читать исходники C#, чтобы понять API — достаточно открыть `/swagger`.

**28. Environment: Development, Staging, Production.**

| Окружение | Где | Настройки |
|-----------|-----|-----------|
| **Development** | Локальная машина | DEBUG=True, БД на localhost, подробные ошибки |
| **Staging** | Тестовый сервер | Максимально похоже на production |
| **Production** | Боевой сервер | Оптимизация, безопасность, мониторинг |

В проекте: `appsettings.json` — для Docker (Host=postgres), `appsettings.Development.json` — для локальной разработки (Host=localhost).

### 0.7. Git и командная работа

**29. Система контроля версий. Зачем одному?**

VCS отслеживает историю изменений. Даже в одиночку: откат к рабочей версии (`git checkout`), эксперименты в ветках без риска, история изменений (`git log` + `git blame`).

**30. Git vs GitHub.**

- **Git** — программа для контроля версий, работает локально. Создана Линусом Торвальдсом в 2005.
- **GitHub** — веб-сервис для хранения Git-репозиториев в облаке. Добавляет Pull Requests, Code Review, Issues, Actions.

**31. Что такое коммит? «Хороший коммит».**

Коммит — снимок состояния файлов. Хороший коммит: маленький (одно логическое изменение), содержательное сообщение (`feat: добавить обработку пустого ввода`), без мусора. Конвенция: `тип: описание` (feat, fix, refactor, docs).

**32. Ветка (branch). Зачем `feature/hash-calc`?**

Ветка — независимая линия разработки. `feature/hash-calc` — ветка для утилиты «Хеш-калькулятор». Пока студент пишет код, `main` стабилен. После завершения: Pull Request → code review → слияние.

**33. Pull Request и code review. Зачем ревьювить?**

Pull Request — запрос на вливание изменений в `main`. Code Review — проверка кода другим разработчиком: ошибки, архитектура, читаемость, безопасность. «Оно работает» ≠ «код хороший».

### 0.8. Безопасность (общий кругозор)

**34. HTTPS vs HTTP. Нужен ли на localhost?**

**HTTPS** = HTTP + шифрование (TLS/SSL). Данные зашифрованы — никто по пути не может прочитать или подменить. **HTTP** — открытый текст, уязвим для перехвата в публичных сетях. На `localhost` HTTPS не обязателен (данные не покидают машину), но рекомендуется для соответствия production.

### 0.9. Тестирование и качество кода

**35. Unit-тест. Зачем, если можно «руками»?**

Автоматическая проверка одного модуля в изоляции. Повторяемость (1000 запусков за секунды), регрессия (сразу видно, что сломалось), документация (тесты показывают ожидаемое поведение), смелость рефакторинга (тесты прошли → всё ок).

**36. «Чистый код». Три признака.**

1. **Читаемость**: понятные имена, нет магических чисел.
2. **Маленькие методы**: один метод — одна задача.
3. **Отсутствие повторений (DRY)**: одинаковый код не копипастится, выносится в отдельный метод/класс.

**37. Рефакторинг vs исправление бага.**

- **Рефакторинг**: меняю внутреннюю структуру, НЕ меняя поведение. Цель — улучшить читаемость.
- **Исправление бага**: меняю поведение (было неправильно → стало правильно).
Золотое правило: не делать рефакторинг и фикс бага в одном коммите.

## Блок 1 — Общая архитектура

### 1.1. База

**38. Из каких трёх основных слоёв состоит приложение?**
- **Frontend** (клиент) — HTML/CSS/JS в браузере, SPA на чистом JavaScript, раздаётся через Nginx.
- **Backend** (сервер приложений) — ASP.NET Core Web API на C#, бизнес-логика утилит, работа с БД.
- **База данных** (хранилище) — PostgreSQL, список утилит и история выполнений.

**39. Что такое SPA и как это реализовано в нашем проекте?**
SPA — веб-приложение, где навигация без перезагрузки страницы. В проекте: `index.html` загружается один раз, содержимое `<main id="app-content">` динамически заменяется через JS. Навигация через хэш-роутер (`#dashboard`, `#utility/sum-numbers`). При смене хэша `Router` вызывает соответствующий рендер-метод.

**40. Чем REST API отличается от классического серверного рендеринга?**
- **Серверный рендеринг**: сервер возвращает готовую HTML-страницу с данными. Каждый переход — полная перезагрузка.
- **REST API**: сервер возвращает только данные (JSON), клиент сам рендерит интерфейс. Разделение ответственности.

**41. Почему frontend и backend в разных Docker-контейнерах?**
- Независимое масштабирование (3 экземпляра backend, 1 frontend)
- Изоляция (обновление frontend не требует пересборки backend)
- Разные технологии (оптимальный базовый образ для каждого)
- Безопасность (backend может быть во внутренней сети)

### 1.2. Средний уровень

**42. Схема: от `http://localhost:3000` до списка утилит.**
```
Браузер → GET localhost:3000 → Nginx → отдаёт index.html
Браузер → загружает CSS/JS → Nginx → отдаёт файлы
Браузер → выполняет app.js → Router → DashboardPage.render()
  → fetch('/api/utilities') → Nginx (location /api/) → api:8080
    → UtilitiesController.GetAll() → AppDbContext.Utilities.ToListAsync()
      → PostgreSQL: SELECT * FROM "Utilities" ORDER BY "Id"
    ← JSON-массив утилит
  → DashboardPage рендерит HTML карточек в #app-content
```

**43. Почему API-запросы идут на `/api/...`, а не на `http://api:8080/api/...`?**
В `config.js`: `apiBaseUrl: '/api'` — относительный путь. Браузер делает запрос к тому же хосту (`localhost:3000`). Nginx видит путь `/api/` и по правилу `location /api/` проксирует в `api:8080`. Если бы использовался `http://api:8080`, браузер пытался бы обратиться к хосту `api`, недоступному извне Docker-сети.

**44. Что такое CORS? Почему `AllowAnyOrigin`? Опасность?**
CORS — механизм браузера, запрещающий JS с одного домена делать запросы к другому без специальных заголовков. В проекте `AllowAnyOrigin` — потому что в разработке фронтенд и backend на разных портах. Опасность в production: любой сайт сможет делать запросы к вашему API. Правильно — указать конкретный домен.

### 1.3. Сложный

**45. Production: домен + HTTPS. Что менять?**
`docker-compose.yml`: убрать `ports` у api (не должен быть доступен снаружи). `nginx.conf`: server на 80 с редиректом на 443, server на 443 с `ssl_certificate`. `Program.cs`: заменить `AllowAnyOrigin` на `WithOrigins("https://utils.company.ru")`, настроить `KnownNetworks`.

**46. Отказоустойчивость: что при падении контейнеров?**
- **Падение `api`**: `restart: on-failure` перезапустит. Nginx вернёт 502. Статика продолжает работать.
- **Падение `postgres`**: `restart: on-failure` перезапустит. API-запросы будут падать с ошибкой подключения. Данные сохранятся в volume. При поднятии БД — healthcheck отработает, api восстановит соединение.

### 2.1. База

**47. Что делает Program.cs? Перечисли ВСЕ шаги.**
1. Создаёт `WebApplicationBuilder`
2. Настраивает `ForwardedHeadersOptions`
3. Регистрирует `AppDbContext` с PostgreSQL
4. Регистрирует `UtilityOrchestrator` (Singleton)

12. `EnsureCreated()` + seed-данные
13. Включает CORS, Swagger, маппит контроллеры
14. Запускает приложение

**48. Что такое Dependency Injection? Пример из проекта.**
DI — паттерн, при котором класс получает зависимости извне через конструктор. Пример: `UtilitiesController(AppDbContext db, UtilityOrchestrator orchestrator)`. Контейнер сам создаёт и передаёт эти объекты. Плюсы: лёгкая замена реализации, упрощение тестирования.

**49. Зачем `IUtilityService`? Почему не базовый класс?**
Интерфейс определяет контракт (Endpoint + Execute), не навязывая реализацию. Базовый класс — жёсткая связь. Интерфейс позволяет DI-контейнеру найти все реализации через `GetServices<IUtilityService>()`.

**50. Маршрут `[Route("api/[controller]")]`. Какой URL у `GetAll()`?**
`[controller]` заменяется на имя контроллера без «Controller». `UtilitiesController` → `utilities`. `GetAll()` с `[HttpGet]` без шаблона → наследует базовый маршрут. Итог: `GET /api/utilities`.

**51. Что такое DTO? Пример из проекта.**
DTO (Data Transfer Object) — объект для передачи данных, не привязанный к модели БД. `ExecuteRequest` принимает `{"input":"..."}`, `ExecuteResponse` возвращает `Output`, `Success`, `Error`. Отделяет формат API от модели данных.

### 2.2. Средний уровень

**52. Как работает UtilityOrchestrator? Почему Singleton?**
Хранит `Dictionary<string, IUtilityService>`: endpoint → сервис. При старте находит все реализации и регистрирует. При вызове `Execute(endpoint, input)` ищет сервис и вызывает его. Singleton — потому что состояние (словарь) неизменно после старта и не зависит от запроса.

**53. Полный путь запроса `POST /api/utilities/sum-numbers/execute`.**
1. Nginx → проксирует в `api:8080`
2. ASP.NET Core: роутинг матчит `UtilitiesController.Execute`
3. Model Binding десериализует тело в `ExecuteRequest`
4. Контроллер ищет утилиту в БД по endpoint
5. `_orchestrator.Execute("sum-numbers", input)`
6. Оркестратор находит `SumNumbersService`, вызывает `Execute`
7. Результат → `ExecuteResponse`, сохраняется в БД
8. `200 OK` с JSON-ответом

**54. Что такое Entity Framework Core? Роль AppDbContext.**
EF Core — ORM: позволяет работать с БД через C#-объекты. `AppDbContext` описывает таблицы (`DbSet<Utility>`, `DbSet<ExecutionResult>`) и связи (`OnModelCreating`). Через него выполняются все запросы: `_db.Utilities.ToListAsync()` → `SELECT * FROM "Utilities"`.

**55. `EnsureCreated()` vs `Migrate()`.**
- `EnsureCreated()` — создаёт БД и таблицы, если их нет. Не отслеживает изменения схемы. Подходит для прототипов.
- `Migrate()` — применяет миграции, позволяет обновлять БД без потери данных.
В проекте `EnsureCreated()` — схема фиксирована на время практики.

**56. Связь Utilities ↔ ExecutionResults.**
В коде (`OnModelCreating`):
```csharp
builder.Entity<ExecutionResult>()
    .HasOne(e => e.Utility).WithMany()
    .HasForeignKey(e => e.UtilityId);
```
В БД: `ExecutionResults.UtilityId` → внешний ключ к `Utilities.Id`. Один Utility — много ExecutionResult.

### 2.3. Сложный

**57. Одновременный вызов из 10 вкладок — гонка данных?**
Гонки данных **нет**: каждый HTTP-запрос в своём потоке, `SumNumbersService` — stateless (нет изменяемых полей), метод `Execute` чистый. Конкурентная запись в БД корректна (транзакции PostgreSQL, автоинкрементные ID).

**58. Фоновая задача: очистка истории старше 30 дней.**
Использовать `BackgroundService` (IHostedService):
```csharp
public class HistoryCleanupService : BackgroundService
{
    protected override async Task ExecuteAsync(CancellationToken ct)
    {
        while (!ct.IsCancellationRequested)
        {
            using var scope = _scopeFactory.CreateScope();
            var db = scope.ServiceProvider.GetRequiredService<AppDbContext>();
            await db.ExecutionResults
                .Where(r => r.ExecutedAt < DateTime.UtcNow.AddDays(-30))
                .ExecuteDeleteAsync(ct);
            await Task.Delay(TimeSpan.FromHours(1), ct);
        }
    }
}
```

## Блок 3 — Frontend (JavaScript SPA)

### 3.1. База

**59. Навигация без перезагрузки. Что такое хэш-роутер?**
Хэш-роутер использует часть URL после `#`. Браузер не отправляет хэш на сервер и не перезагружает страницу. В проекте: `Router` слушает `hashchange`, вызывает обработчик, страница рендерит HTML в `#app-content`. Прямая ссылка `/#utility/sum-numbers` → загружается `index.html`, JS читает хэш и рендерит страницу.

**60. Как работает Router.navigate()? Что при несуществующем хэше?**
1. Берёт хэш, убирает `#`, если пусто → `'dashboard'`
2. Для каждого маршрута создаёт regex, ищет совпадение
3. Нашёл → вызывает handler с параметрами
4. **Не нашёл** → вставляет HTML «Страница не найдена» со ссылкой на дашборд

**61. Что делает DashboardPage.render()?**
1. Показывает индикатор загрузки
2. `API.get('/utilities')` → запрос к backend
3. Собирает категории, строит HTML: заголовок, фильтры, карточки
4. Вставляет в DOM, навешивает обработчики: фильтры → `filter()`, карточки → `location.hash`

**62. Как frontend узнаёт, реализована утилита? Где обновляется?**
`Utility.IsImplemented` устанавливается **в контроллере**: оркестратор возвращает список зарегистрированных endpoint'ов, контроллер сверяет с БД и проставляет `IsImplemented = true/false`. Frontend просто читает поле из JSON. При добавлении сервиса в DI и перезапуске — флаг обновляется.

**63. Зачем нужна UtilityPage.escape()? Что без неё?**
Защита от XSS: экранирует `<`, `>`, `&`, `"`. Без неё `<script>alert(1)</script>` в истории выполнений выполнился бы в браузере. С escape: превращается в `&lt;script&gt;` и отображается как текст.

### 3.2. Средний уровень

**64. Почему скрипты без defer/async?**
Скрипты в конце `<body>` → DOM уже загружен к моменту выполнения. Порядок важен: `config.js` → `api.js` → `router.js` → pages → `app.js`. Без `defer`/`async` выполняются строго последовательно.

**65. Фильтрация без повторного запроса. Плюсы и минусы.**
Фильтрация клиентская: скрытие/показ DOM-элементов через `display`. Плюсы: мгновенно. Минусы: не работает с пагинацией, при 1000+ элементах страница тяжелеет. Для production — серверная фильтрация.

**66. Почему addEventListener после вставки HTML?**
`querySelectorAll` ищет элементы в текущем DOM. Если навесить обработчики до `innerHTML = html`, элементов ещё нет — пустой NodeList. Порядок: `innerHTML` (элементы в DOM) → `querySelectorAll` (находим) → `addEventListener` (навешиваем).

## Блок 5 — Docker и инфраструктура

### 5.1. База

**77. Multi-stage build в backend-Dockerfile.**
```dockerfile
FROM mcr.microsoft.com/dotnet/sdk:8.0 AS build   # SDK (~800 MB) — компиляция
COPY *.csproj . && dotnet restore
COPY . . && dotnet publish -c Release -o /app

FROM mcr.microsoft.com/dotnet/aspnet:8.0          # runtime (~220 MB) — запуск
COPY --from=build /app .
```
Зачем: итоговый образ без SDK и исходников — меньше размер, быстрее загрузка, меньше поверхность атаки.

**78. Healthcheck postgres. Что без него?**
`pg_isready` проверяет готовность PostgreSQL. Без healthcheck: `api` стартует сразу после контейнера `postgres`, но БД ещё не готова → ошибка подключения. `depends_on: condition: service_healthy` гарантирует порядок.

**79. Как контейнеры общаются? Сеть intern-net.**
Docker создаёт виртуальную сеть `intern-net` (bridge). Каждый контейнер получает IP и DNS-имя = имя сервиса. `api` → `http://postgres:5432`, `frontend` → `http://api:8080`. Связь только внутри сети; наружу — только явные `ports:`.

### 5.2. Средний уровень

**80. Порт PostgreSQL наружу. Нужен ли в production?**
`"5432:5432"` — удобно для разработки (DBeaver/pgAdmin). В production — нет: БД не должна быть доступна снаружи. Достаточно внутренней docker-сети.

**81. `docker-compose up -d --build` — по шагам.**
1. `--build` — пересобрать образы (api, frontend)
2. Создать сеть `intern-net`, volume `postgres_data`
3. Запустить `postgres`, ждать `healthy`
4. Запустить `api` (после healthy)
5. Запустить `frontend` (после api)
6. `-d` — detached mode (в фоне)

**82. `.dockerignore` — зачем?**
Исключает файлы из контекста сборки (`bin/`, `obj/`, `node_modules/`, `.git/`). Без него — огромный размер, долгая передача контекста, кэш-промахи. Уменьшает контекст в 10-100 раз.

**83. Уменьшение backend-образа.**

## Блок 8 — Безопасность (общий кругозор)

**98. SQL-инъекции. Как предотвращено?**
EF Core использует параметризованные запросы: `_db.Utilities.FirstOrDefaultAsync(u => u.Endpoint == endpoint)` → `WHERE "Endpoint" = @p0`. Даже если `endpoint = "'; DROP TABLE--"`, это будет просто значение параметра.

**99. XSS-защита в проекте.**
В `utility.js` функция `UtilityPage.escape()`:
```javascript
escape(str) {
    return String(str)
        .replace(/&/g, '&amp;').replace(/</g, '&lt;')
        .replace(/>/g, '&gt;').replace(/"/g, '&quot;');
}
```
Применяется ко всем пользовательским данным перед `innerHTML`. Без неё `<script>alert(1)</script>` выполнился бы.

**100. X-Forwarded-* заголовки. Уязвимость без них.**
Передают реальный IP клиента, протокол, префикс прокси через reverse proxy. Без них: Swagger построит неправильный base URL, Try It Out не сработает. Уязвимость: если не очищать `KnownProxies`, злоумышленник может подделать `X-Forwarded-For`. В проекте `KnownNetworks.Clear()` — в production нужно указать конкретную сеть.

**101. Пароль в appsettings.json — допустимо ли?**
В учебном проекте — да (БД только локально). В production — нет. Правильно: переменные окружения (`${DB_PASSWORD}` из `.env` в `.gitignore`), Docker secrets, Azure Key Vault / AWS Secrets Manager, `dotnet user-secrets` в разработке.
